## 📌 Role: Enhanced Code Quality for `pyear`

This document guides AI agents (e.g., OpenAI Codex, ChatGPT) in generating consistently high-quality, production-ready code aligned precisely with the standards of the `pyear` project, which specializes in fatigue detection and analysis of blink-related ocular signals.

---

## 🧠 Project Overview

* **Purpose**: Analysis of eye aspect ratio (EAR)  data for fatigue detection.
* **Structure**: Modular, clearly organized by functional domains.
* **Primary Interface**: `pipeline.extract_features()`
* **Data Input**: Supports `mne.Epochs` (segmented data) and `mne.Raw` (continuous signals).
* **Output**: Features extracted as structured `pandas.DataFrame`.

## 📌 Key Coding Practices

### 📄 Input Handling and Testing

* **Assumptions**:

    * All input data is either:

        * `mne.Epochs` (segmented into fixed-length epochs), or
        * `mne.Raw` (continuous signals).
    * Detection logic must identify and appropriately handle each case clearly.
    * Utilize common functions wherever feasible; however, aggregate epoch-specific results separately.

* **Testing Inputs**:

    * Mock EAR signal generation provided via `unittest.fixtures.mock_ear_generation`
    * Unit tests must thoroughly cover both continuous and epoch scenarios

### 🪵 Logging Standards

* Define root logger configuration in `pipeline.py`
* Each Python module uses: `logger = logging.getLogger(__name__)`
* Top-level functions include:

    * `INFO` logs indicating function entry/exit
    * `DEBUG` logs capturing crucial variable states and decision points

### ⏳ Progress Feedback

* Consistently use progress bars (`from tqdm import tqdm`) for loops handling substantial iterations (e.g., epochs or signal segments).

### 📄 Docstrings and Type Hints

* Use clear, structured docstrings in Google or reStructuredText format.
* Explicitly document parameters, return types, and exceptions.
* Type annotations (`typing`) are mandatory for function signatures.

### 🧪 Comprehensive Unit Testing

* Implement comprehensive testing with:

    * `pytest` or Python's built-in `unittest`.
    * Clearly segregated test modules under the `unittest/` directory.
    * Unit tests leveraging synthetic EAR signals generated by provided mocks.
